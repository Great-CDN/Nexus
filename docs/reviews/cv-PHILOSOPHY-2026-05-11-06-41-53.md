## Detailed Reviews

### DeepSeek (deepseek-chat)

Here is a technical review of the provided `PHILOSOPHY.md` document.

- **Issue: Ambiguity in "Approved by Human" timeline.** The document states "Acceptance is human" and "AI never commits, deploys, or merges without human approval." However, it does not specify *when* in the workflow this approval occurs relative to other verification steps (e.g., automated tests, linters). This ambiguity could lead to a human approving code that then fails automated checks, or an automated system deploying code that bypassed human review.
    - **Severity:** Medium
    - **Suggested Fix:** Clarify the approval sequence. For example: "Human approval must occur **after** all automated checks (linting, tests, security scans) have passed. A human can only approve a commit or merge request that has a clean CI/CD status."

- **Issue: Contradiction on "High Earth Orbit Correction" for small projects.** The document correctly states that "Explicit Context > Implicit Memory" works for small projects. The `docs/SCALE.md` reference for large projects is helpful, but the philosophy itself lacks a concrete, actionable rule for what "small" means and what the transition looks like. A developer might not realize they've crossed the threshold until they hit a failure.
    - **Severity:** Low (for this document, but critical for the overall system)
    - **Suggested Fix:** Add a quantitative or qualitative definition for "small project" within this context. Example: "A project is 'small' for the purposes of explicit context loading if all critical code, specs, and designs can fit within a single session's context window (e.g., < 100K tokens). When this is no longer true, you must switch to a module-contract-and-layered-abstraction approach, as described in `docs/SCALE.md`."

- **Issue: The rule "Small Context > Large Context" threshold is too specific and arbitrary.** Stating "1200 lines is the absolute maximum before the model begins silently truncating context" is a significant technical error. Context limits are measured in tokens (e.g., 128K, 200K), not lines of code. 1200 lines of a configuration file is tiny; 1200 lines of a complex algorithm with long comments could be large. This misstatement undermines the credibility of the entire section.
    - **Severity:** High
    - **Suggested Fix:** Replace line counts with a token-count-based heuristic. Example: "Your context package (spec + design + relevant code) should not exceed 50,000 tokens. If it does, the task is too large and must be split. A good rule of thumb is that a context package exceeding ~500-800 lines of human-written spec/documentation is too large; for generated code, use the token limit as the hard boundary."

- **Issue: Missing guardrail for "Pattern Expansion" capability.** The document correctly identifies "Pattern expansion" as an AI strength. However, it doesn't account for a dangerous variation: *hallucinatory pattern expansion*. The AI might generate plausible-looking code that follows a pattern but includes subtle errors like off-by-one errors, incorrect function signatures, or insecure default values in repeated boilerplate.
    - **Severity:** Medium
    - **Suggested Fix:** Add a qualifying statement to the "Pattern expansion" bullet. Example: "**Pattern expansion**: given an example, generate more of the same. **Risk:** The model may invent non-existent variations of the pattern or apply it to contexts where it's invalid. Human review is required to verify each generated instance against the original pattern and its specific context."

- **Issue: The "Review > Generation" section underestimates AI's role as a reviewer.** The document states that human review is for "what machines cannot judge." While this is true for architecture and business logic, modern LLMs *can* be effective reviewers for certain aspects that humans are poor at, such as: finding logic errors in loops, checking for off-by-one errors, or detecting spelling/grammar inconsistencies in code comments. The document's binary "AI generates / Human judges" framework is insufficiently nuanced for a modern workflow.
    - **Severity:** Low
    - **Suggested Fix:** Acknowledge that AI can be a *first-pass reviewer*. Example: "Human review is for what machines *ultimately* cannot judge. However, use AI as a **first-pass reviewer** to catch obvious logic errors, style inconsistencies, and format violations before human review begins. This reduces human fatigue and allows humans to focus on higher-value judgments. The human still retains final authority."

- **Issue: The "Anti-Goals" section lists "Hidden memory systems" but doesn't address "Hidden tool usage."** The document correctly bans opaque state, but it does not address a modern anti-pattern: AI agents that silently call external tools (e.g., running a script, making an API call, modifying a file) without explicit human confirmation for each action. This is a significant security and control gap.
    - **Severity:** High
    - **Suggested Fix:** Add to the Anti-Goals list: "**Hidden tool execution**: AI must not execute any tool (run command, edit file, call API) without explicit, approved human instruction for that specific action. No 'agentic' auto-fixing."
