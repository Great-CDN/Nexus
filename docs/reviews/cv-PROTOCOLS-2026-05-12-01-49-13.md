## Detailed Reviews

### DeepSeek (deepseek-chat)

- **Issue 1: Ambiguity in “CLAUDE.md §Critical Rules (Absolute)”**  
  **Severity:** High  
  **Description:** The priority table references `CLAUDE.md §Critical Rules (Absolute)` but does not define which document this is or where it lives. If multiple `CLAUDE.md` files exist (e.g., at repo root vs. subdirectory), or if the document is renamed, the priority rule becomes unenforceable.  
  **Suggested Fix:** Specify the exact path to `CLAUDE.md` (e.g., `./CLAUDE.md`) and clarify whether only the `§Critical Rules` section has priority 1, or the entire document. Add a note that `CLAUDE.md` must be present for the hierarchy to function.

- **Issue 2: Conflict resolution language is incomplete for priority ties**  
  **Severity:** Medium  
  **Description:** The table assigns unique priorities (1-8), but if a document is accidentally omitted or mislabeled, a tie could occur. The text does not explain how to resolve conflicts between documents with the same priority (e.g., two templates in different paths).  
  **Suggested Fix:** Add a tie-breaking rule: “If two documents share the same priority level, the document modified more recently wins. If timestamps are equal, the more specific path (deepest subdirectory) wins.”

- **Issue 3: Template conflict resolution delegates to Triage Protocol, but Triage Protocol may depend on templates**  
  **Severity:** Medium  
  **Description:** The statement “the Triage Protocol decision (Full vs. Light Workflow) determines which [template] applies” creates a circular dependency if the Triage Protocol itself uses a template to define its decision logic.  
  **Suggested Fix:** Explicitly state that the Triage Protocol file (`docs/protocols/triage.md`) is **not** inside `docs/templates/` and that its content is normative over any template decision. Add: “Template selection rules must be defined in a protocol (≥ priority 2), not in templates themselves.”

- **Issue 4: Examples of invalid overrides are incomplete**  
  **Severity:** Low  
  **Description:** The three examples given all illustrate a lower-priority document overriding a higher one. They do not cover cases where a higher-priority document makes an ambiguous statement, and a lower-priority document could clarify without contradicting.  
  **Suggested Fix:** Add a note: “If a higher-priority document is ambiguous, a lower-priority document may provide interpretation, but the interpretation must be consistent with all higher-priority documents. Any contradiction renders the lower-priority document invalid.”

- **Issue 5: Directory table missing a “Status” column**  
  **Severity:** Low  
  **Description:** The protocol directory lists names and purposes but does not indicate whether each protocol is final, draft, or deprecated. Over time, protocols may be updated or retired without clear tracking.  
  **Suggested Fix:** Add a `Status` column with possible values: `Active`, `Draft`, `Deprecated`. Require that all protocols in `docs/protocols/` have a status header.

- **Issue 6: No recovery mechanism for protocol violations**  
  **Severity:** High  
  **Description:** The document states “Violating a protocol means the output is invalid until corrected” but does not describe how to detect violations, who corrects them, or what the fallback process is when a violation is detected mid-execution.  
  **Suggested Fix:** Add a subsection titled “Violation Handling” that specifies: (a) automated validation checks at session start, (b) a required review step after any protocol violation, (c) a path to re-issue corrected output, and (d) an escalation rule to stop execution if a Critical protocol is violated.
