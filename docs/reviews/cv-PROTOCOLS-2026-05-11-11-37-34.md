# Technical Review of PROTOCOLS.md

## Issues Found

### Critical

1. **Missing lock file race condition handling for atomic writes**
   - **Description**: Section Memory Protocol > Memory Maintenance states that for team environments with concurrent writers, a lock file should be considered ("If a lock is not feasible, accept the risk"). Accepting risk without mitigation for concurrent writes means corruption can still occur. The atomic write pattern (write to temp, then rename) is not sufficient when two processes write to different temp files and both renames succeed; the last write wins, potentially losing data.
   - **Severity**: Critical
   - **Suggested Fix**: Either mandate the lock file for any team environment, or change the atomic write to use a versioned sequence (e.g., `memory_01.md`, `memory_02.md`) and have readers read all files, merging by recency. If the environment cannot support locks, define a mitigation strategy (e.g., comparison of mtime + content hash).

2. **Context Packaging Protocol line limits are internally contradictory**
   - **Description**: The protocol states "Keep it under 800 lines ideally, 1200 lines absolutely" but then adds "These numbers are heuristics based on typical token density, not hard limits." The Threshold Classification table classifies 1200 lines as a Hard Limit ("Process is invalid; split the task immediately") and 800 lines as a Soft Limit. The Context Packaging Protocol calls 1200 a heuristic, but Threshold Classification calls it a Hard Limit. These are contradictory.
   - **Severity**: Critical
   - **Suggested Fix**: Reconcile the two sections. Either make 1200 a Hard Limit consistently across both documents (and remove "heuristics" language from Context Packaging), or classify it as Soft Limit in Threshold Classification and remove the Hard Limit classification. The paragraph about token estimation also weakens the rule—either the line count is a limit or it is not.

3. **Session Recovery Protocol step "Preserve uncommitted output" has unsafe write destination**
   - **Description**: The protocol says to save uncommitted output to `.claude/scratch/YYYY-MM-DD-HHMM-<task>.md`. It does not specify atomic write or locking for this file. If a crash occurs during the write of the scratch file itself (e.g., partial write), the saved state is corrupt and the breakpoint data is lost.
   - **Severity**: Critical
   - **Suggested Fix**: Use the same atomic write pattern (write to temp, rename) for scratch files. Also, add a step to verify the scratch file can be read back before proceeding with recovery.

4. **No validation mechanism for "Spec is Complete"**
   - **Description**: The Spec Format Protocol lists mandatory sections but provides no automated or human-checkable way to verify that the spec is complete before moving to Design. The Review Protocol's Design Review Checklist checks completeness against acceptance criteria, but there is no gate for the spec itself. A spec missing one mandatory section (e.g., "Open Questions") could pass into Design.
   - **Severity**: High
   - **Suggested Fix**: Add a pre-Design gate: "Before Design phase begins, run the Spec Completeness Checklist: all 7 mandatory sections present and non-empty; all acceptance criteria have verifiable metrics or Given/When/Then format; open questions have owners defined within 1 business day." This could live in CHECKLISTS.md with a reference from PROTOCOLS.md.

5. **Commit Protocol: "One task, one commit" contradicts "Light Workflow exception"**
   - **Description**: The Commit Protocol Rule 1 says "One task, one commit" with a Light Workflow exception allowing a single commit for multiple trivial changes. However, the Triage Protocol says Light Workflow is for "small enhancements, UI tweaks, adding a field to a form, routine dependency updates." The exception permits bundling changes, but nothing defines how to separate "trivial, related changes" from "unrelated changes that should be separate commits." This opens a loophole for large bundling.
   - **Severity**: High
   - **Suggested Fix**: Add a concrete rule: "The Light Workflow exception permits bundling only when all changes touch the same file(s) and the total diff is under 50 lines. Otherwise, split into separate commits." This prevents abuse.

6. **API Failure Classification: "Retry with exponential backoff" for transient errors is underspecified**
   - **Description**: The protocol says "Retry with exponential backoff: 1s, 2s, 4s. Max 3 attempts." It does not specify jitter. Without jitter, multiple concurrent sessions (or multiple users in a team) hitting the same rate limit will retry in lockstep, potentially causing a thundering herd and continued failures.
   - **Severity**: High
   - **Suggested Fix**: Add jitter: "Retry with exponential backoff + random jitter: base delays 1s, 2s, 4s, each multiplied by a random value in [0.5, 1.5). Max 3 attempts." This is a best practice for distributed rate-limited APIs.

7. **Missing specification for how to handle concurrent memory writes in the "last write wins" model**
   - **Description**: The Memory Protocol states "If two sessions write simultaneously, the last write wins. Use descriptive filenames to avoid collisions." This is not sufficient for a team environment. Two sessions could write different content to the same filename (e.g., `user_prefers_tabs.md`) and the last write wins, losing data. Descriptive filenames only avoid same-filename collisions by chance; they are not a collision-prevention mechanism.
   - **Severity**: High
   - **Suggested Fix**: Either: (a) require the lock file for all writes to memory (remove the "accept the risk" option), (b) mandate collision-resistant naming via timestamp + user ID (e.g., `2026-05-11-alice-prefers-tabs.md`), or (c) require a content-merge step during memory maintenance.

8. **Threshold Classification: Soft Limits lack enforcement mechanisms**
   - **Description**: The table says Soft Limits require "explicit justification" and "document why the exception is safe." But there is no defined location or format for this justification. Without a standard place (e.g., a comment in the task tracker, a note in the commit message, a justification section in the design doc), the requirement is unenforceable and will be ignored in practice.
   - **Severity**: Medium
   - **Suggested Fix**: Add a rule: "Soft Limit justifications must be documented in the task's design doc or a comment in the spec under a new section labeled 'Soft Limit Override Justification.' The justification must state which limit is exceeded, by how much, why the exception is safe, and what monitoring is in place." This makes it enforceable.

9. **Artifact Quality Standard: Line limits for mixed-content vs. pure prose are inconsistent with earlier rules**
   - **Description**: The Standard says mixed-content artifacts must not exceed 800 lines, and pure prose may reach ~2000 lines. However, the Context Packaging Protocol says mixed content should be under 800 lines (soft) and 1200 lines (absolute/hard limit depending on which section you read). The Standard does not define where the Spec Format Protocol's "800-line mixed-content limit" fits: is a spec with embedded code snippets "mixed-content" subject to 800 lines, or "pure prose" up to 2000 lines? This ambiguity can lead to violations.
   - **Severity**: Medium
   - **Suggested Fix**: Define "mixed-content" precisely: any artifact containing code blocks (fenced code or inline code representing implementation, not just short examples) exceeds X% code by line count? Alternatively, explicitly state that specs with embedded code blocks are mixed-content and subject to the 800-line limit; specs with no code blocks are pure prose and subject to 2000 lines.

10. **Review Protocol: "Automated checks must pass before human review" creates a chicken-and-egg problem for missing tooling**
    - **Description**: The Fallback says "If a required automated check is not available... document the gap... and manually verify that check's concern during human review. The goal is to have all checks operational before the next review cycle." This is problematic because: (a) "before the next review cycle" could be interpreted as the same cycle if checks are quickly added, but the fallback suggests manual verification in the meantime; (b) there is no mechanism to enforce that the gap is actually closed; (c) if a project does not have any security scanner configured, the human reviewer must manually verify all security concerns, which is impractical for large changes.
    - **Severity**: Medium
    - **Suggested Fix**: Add a concrete timeline: "Gaps in required checks must be resolved within 2 business days or before the next review cycle, whichever is longer. If a check is missing for more than 30 days, escalate to the project lead." Also, provide a template for the manual verification record (e.g., "Security review performed manually on [date]; issues found: none / [list]").

11. **Spec Versioning: "During Design: if the spec needs revision to make design possible, bump to v1.1" ambiguous**
    - **Description**: The protocol says to bump to v1.1 during Design if the spec needs revision. But v1.1 is defined as "Minor revisions (clarifications, typo fixes, additional examples) that do not change scope or acceptance criteria meaning." If the revision makes design possible, it likely changes the spec meaning (e.g., clarifying a previously ambiguous acceptance criterion could change its interpretation). So v1.1 may actually be v2.0, but the protocol does not provide guidance for this grey area.
    - **Severity**: Medium
    - **Suggested Fix**: Add a clarification: "If a revision during Design changes how an acceptance criterion would be verified (even if the criterion's text is unchanged), it must be a v2.0 revision. If it only adds examples or corrects typos without altering any possible interpretation, it is v1.1. When in doubt, choose v2.0."

12. **Missing protocol for spec deprecation/removal**
    - **Description**: The Spec Versioning protocol describes how to create new versions but not how to deprecate or remove a spec. If a spec is abandoned (feature canceled), there is no guidance on how to mark it as superseded or removed.
    - **Severity**: Low
    - **Suggested Fix**: Add a rule: "When a spec is superseded or a feature is canceled, update the spec's frontmatter with `status: superseded` or `status: canceled` and include a reference to the replacement spec or the cancellation decision." This prevents confusion from orphaned files.

13. **Definitions: Severity Levels include "Severity assignment must include the justification" but no enforcement**
    - **Description**: The Severity Levels table has a "Justification Required" column, and the text says "Every severity assignment must include the justification described in the Justification Required column. A severity without justification is incomplete." But there is no verification step (no checklist item, no automated check) to ensure justifications are provided.
    - **Severity**: Low
    - **Suggested Fix**: Add a bullet to the Artifact Quality Standard's "Complete" dimension: "Every severity assignment (Critical, High, Medium, Low) includes its required justification per Definitions." This makes it part of the quality checklist.

14. **Memory Protocol: "Atomic writes: Each memory entry is one file. Do not append to a shared file." contradicts team convention**
    - **Description**: Many team workflows (e.g., Obsidian, Notion, shared Git repos) prefer index files or shared files for discoverability. The protocol explicitly forbids appending to shared files, which may conflict with team norms and increase cognitive overhead to find memory entries.
    - **Severity**: Low
    - **Suggested Fix**: Add an optional convention: "If the team prefers, maintain an index file (`.claude/memory/INDEX.md`) listing `name`, `type`, `description`, and last-modified date for all active memory entries. This index is regenerated from the files and must not contain content duplicated from individual files." This provides discoverability without violating atomic writes.

15. **Branch Naming: "fix/auth-cache-key" example uses a single category "fix" for all bug fixes regardless of scope**
    - **Description**: The Branch Naming table lists only one fix pattern: `fix/<scope>-<description>`. This does not distinguish between hotfix (production critical) and regular bug fixes. A production hotfix branch named `fix/auth-token-leak` lacks the signal-to-noise distinction that `hotfix/` provides.
    - **Severity**: Low
    - **Suggested Fix**: Add a rule: "Use `hotfix/` only for production-critical fixes that bypass normal review (as defined in Triage Protocol > Hotfix). All other bug fixes use `fix/`." This aligns with the Triage Protocol's hotfix definition.

16. **Missing protocol for merging Pull Requests**
    - **Description**: The Commit Protocol covers commit messages and signing but does not specify merge strategy (squash merge, rebase merge, merge commit). Different strategies have different implications for audit trail quality.
    - **Severity**: Low
    - **Suggested Fix**: Add a rule: "Acceptance-phase merges to `main` must use squash merges for feature branches with multiple WIP commits, or rebase merges for branches with clean commit history. Merge commits are reserved for releases or cross-team coordination." This preserves the "one commit per task" principle.

17. **Definitions for "Critical" severity includes "No workaround" but does not define what constitutes a workaround**
    - **Description**: The Severity Levels table defines Critical as "No workaround." But what counts? A user can manually edit a config file to fix a configuration bug? A data loss bug could be worked around by restoring from a backup—does that count? Without clarity, severity assignments will be inconsistent.
    - **Severity**: Low
    - **Suggested Fix**: Add a footnote: "A workaround is any action the user can take to achieve the expected behavior without deploying new code. For example, restarting the service, editing a config file, or using a different feature path. A workaround is not 'restoring from backup' or 'rolling back to previous version'—those are recovery actions, not workarounds." This clarifies the definition.

18. **Quality Checklist: "No orphaned references" is required but no mechanism to detect orphaned references**
    - **Description**: The Artifact Quality Standard specifies "No orphaned references" as a requirement. However, there is no automated or manual verification mechanism in the checklist to detect if a reference to a file or section exists that no longer exists.
    - **Severity**: Low
    - **Suggested Fix**: Add a verification step: "Run a broken-link checker over Markdown files (e.g., `mlc` or `markdown-link-check`) before marking the artifact complete. Document any false positives (e.g., references to files created in the same changeset) in the review notes." This makes the requirement actionable.
