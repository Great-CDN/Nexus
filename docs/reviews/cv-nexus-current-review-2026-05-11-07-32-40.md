## Detailed Reviews

### DeepSeek (deepseek-chat)

Here is the technical review of the provided artifact.

- **High: Spec Format Protocol §Spec Rules allows for a dangerous interpretation.** The rule states "One spec per feature" and "Spec changes must be explicit... Do not silently deviate from the approved spec." This is correct for bounded features, but ambiguity arises when a single acceptance criterion (AC) is too large or when a feature is decomposed. The rule does not define what constitutes a "feature" vs. a "task," potentially leading to monolithic specs. **Fix:** Clarify that if a single AC requires multiple implementation tasks (from the design breakdown), it is acceptable to have one spec that is implemented over multiple sessions, as long as each task references its specific ACs. The key is to prevent scope creep, not to mandate a 1:1 spec-to-task ratio.
- **High: Checklist CHECKLIST: Implementation requires human to "understand the code," but does not address scenarios where the code is too complex.** The checklist includes: "Human has read and understands the code." This is an unverifiable and potentially dangerous criterion. A human may *think* they understand complex generated code that contains subtle logic errors or security vulnerabilities. **Fix:** Replace "understands the code" with a verifiable process. For example: "Human has traced the execution path for the primary acceptance criterion and the primary error path." or "Human has walked through the code with an AI debugger to verify state transitions." This forces a demonstrable action, not a subjective feeling.
- **High: Workflows Phase 3: Implementation §Exit Conditions requires review before commit, but Light Workflow §Levels suggests Light workflow can skip review for trivial changes.** Phase 3 Implementation checklist item "Human has reviewed and understood the code" is mandatory for Full workflow. The Light workflow description says "Implementation: One task, one session. No formal task template needed." but does not explicitly override the Implementation phase's exit conditions, including the review. **Fix:** In the Triage Protocol §Levels for Light Workflow, explicitly state that the "Human has reviewed and understood the code" requirement from the Implementation checklist is **reduced** to "Human has scanned the diff for obvious errors," while still requiring automated checks (lint, type-check) to pass. This creates a clear, lighter standard.
- **Medium: Cross-Validation §Implementation Guide Step 3: Isolate Context mandates deception that contradicts the "Explicit" value.** The document instructs to present an artifact without attribution ("Here is a design document for review... no attribution"). The stated goal is to avoid leniency or hostility. However, the project's explicit value system (§Engineering Values) mandates "Explicit Over Implicit." Deception, even for a valid reason, is implicit. **Fix:** Add a "Transparency Note" to the protocol. Before the second model reviews, explicitly state: "This artifact was generated by another AI model [Model A]. Your task is to identify any flaws, risks, or ambiguities independent of its source." This is explicit, achieves the goal of removing identity bias, and aligns with the project's stated values. The real bias to remove is "I would have done it this way," not the model's self-attribution.
- **Medium: Metrics §Core Metrics 1. Rework Rate definition conflates different types of failure.** "Tasks with Request Changes verdict" from review encompasses both design clarity issues (upstream) and implementation correctness issues (downstream). A high Rework Rate could mean the implementation is poor, or it could mean the review is overly strict. **Fix:** Break "Rework Rate" into two sub-metrics to improve diagnostic value.
    - **Upstream Rework Rate:** Change requests that require a **spec or design** revision (e.g., "this doesn't solve the problem").
    - **Implementation Rework Rate:** Change requests that fix **code implementation** of a correct spec/design (e.g., "this logic is wrong").
    This provides a clear signal for whether the problem is in the planning or the execution.
- **Medium: Decision Principles §Decision Principles has a severe self-contradiction.** Principle #2: "If the human is not making a decision, the process is too automated." Principle #3: "If AI is doing something you do not understand, stop and review." The contradiction is that by the time a human "does not understand" what AI is doing, a decision has already been delegated to the AI (the decision to proceed without understanding). **Fix:** Rephrase Principle #3 to eliminate the contradiction: "If you cannot explain the purpose of every AI-generated action in the last 15 minutes, the process has violated Principle #2 (human non-decision). Stop and revert the last step." This turns the monitoring requirement into a concrete, actionable check on autonomy.
- **Low: Workflows Phase 4: Testing §Exit Conditions says "No critical or high bugs remain." This is ambiguous about "high" bugs.** The checklist for Testing includes "No Critical or High bugs remain." However, the Definitions in Protocols define "High" as something with a workaround that is "painful." **Fix:** For the Testing phase exit condition, change the criterion to be more restrictive: "No Critical bugs remain. No more than [X] High bugs remain, and they are documented and accepted by the human." This allows for pragmatic shipping decisions while maintaining transparency.
- **Low: Spec Template (docs/templates/spec.md) §Acceptance Criteria uses "verifiable" and "metric-based" examples but is not strict enough.** The template shows examples like "AC-1: <Specific, verifiable criterion>." This is too vague. **Fix:** Mandate a format in the template. Example: `- [ ] AC-1: (Given/When/Then or Metric) <Description>`. Acceptable examples: "Given the user is on the settings page, when they toggle 'Dark Mode' to 'On', then all application text backgrounds turn black." or "The endpoint `/api/report` returns a response in under 200ms for 100 concurrent requests (p95)." This forces precision from the start.
- **Low: Workflows Phase 2: Design §Exit Conditions and §Validation Criteria do not require a security or risk review.** The design phase output includes interface contracts and task breakdown but does not mandate a security or risk analysis as a validation criterion. Security is left to later phases (Testing, Review). **Fix:** Add a validation criterion to the Design phase: "A security/privacy risk assessment has been performed (using docs/CROSS-VALIDATION.md if appropriate) for all interface contracts and data model changes." This is a critical point to catch architectural vulnerabilities.
- **Low: Protocols §Prompt Protocol Rule: "Request one artifact per prompt" is correct but unenforceable for complex tasks.** The rule says "One function, one component, one test file." For a complex task like implementing a 10-function class, requesting 10 separate prompts is inappropriately burdensome. **Fix:** Change the rule to: "Request one coherent unit of work per prompt. A unit is a single function, a single component, a single interface contract, or a single test file. A class with interdependent methods is a unit; do not break it into 10 separate prompts." This allows for logical grouping while preventing open-ended generation.
