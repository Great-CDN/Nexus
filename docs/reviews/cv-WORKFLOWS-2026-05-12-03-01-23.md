## Technical Review: Workflows Document

### Issues Found

- **Critical: Missing fallback for "Run CHECKLIST" without defined checklists.** The document repeatedly references `docs/CHECKLISTS.md` and specific checklists (e.g., `CHECKLIST: Specify`, `CHECKLIST: Design`), but these are not included in the artifact. If these checklists do not exist or are inaccessible, the workflow cannot proceed as defined, creating a hard dependency on missing artifacts. **Severity: Critical** — blocks the entire workflow. **Fix:** Either include the checklists inline (or as appendices), or define the minimal validation criteria directly in this document as a fallback, with a clear note on where to find the canonical checklists.

- **High: "No single task exceeds 2 hours of estimated work" is ambiguous and unverifiable.** The Design phase requires that no task exceeds 2 hours of estimated work, but there is no guidance on how to estimate, who estimates, or what happens if a task naturally exceeds 2 hours (e.g., a complex refactor). This rule can lead to artificially splitting tasks into meaningless sub-tasks just to meet the constraint. **Severity: High** — unenforceable and may degrade design quality. **Fix:** Replace with a relative sizing rule (e.g., "no task should be larger than can be implemented in a single focused session") or provide an explicit estimation guideline (e.g., "use T-shirt sizing: if a task is 'large,' split it"). Add a note: "If a task is unavoidably >2h, document the justification and proceed."

- **High: Exit condition for "Decide" phase includes "No critical technical debt introduced" but provides no definition or measurement.** This is subjective and unenforceable without a shared understanding of what constitutes "critical technical debt." Different engineers will have different thresholds. **Severity: High** — leads to inconsistent decisions. **Fix:** Define "critical technical debt" in terms of concrete criteria (e.g., "any missing error handling, hardcoded secrets, or unhandled failure modes in a security-sensitive path") or reference a separate debt policy. Alternatively, remove the criterion or make it a "flag for discussion" rather than a pass/fail.

- **Medium: Hotfix Workflow is mentioned but never fully defined.** The document references "Hotfix Workflow" as a variant but does not specify its phases, inputs/outputs, or exit criteria. It says "use for: Urgent minimal fixes; Narrow scope; Known issue, known remedy," but users cannot execute it without additional documentation. **Severity: Medium** — incomplete specification of a workflow variant. **Fix:** Add a brief section defining the Hotfix Workflow (even if it's a subset of the Light Workflow with relaxed validation) or include a pointer to where it is defined (e.g., `docs/protocols/hotfix.md`).

- **Medium: "Human can explain what every changed line does" in Execute phase is impractical for large changes.** This validation criterion is reasonable for a single task (which is already constrained to ≤2 hours), but if a task involves many lines (e.g., auto-generated boilerplate or schema migrations), requiring the human to explain every line is burdensome and may be bypassed. **Severity: Medium** — overhead without proportional safety. **Fix:** Clarify: "Human can explain the purpose and logic of every changed region; AI-generated boilerplate, configuration, or repetitive patterns need only be reviewed for correctness, not individually explained."

- **Medium: "Automated checks pass: type check, lint, static analysis, tests" — no specification of which tools or configurations.** Different projects use different linters, type checkers, and static analysis tools. The document assumes a specific toolchain but does not define it. **Severity: Medium** — ambiguity leads to inconsistency. **Fix:** Reference a project-level `pyproject.toml`, `.golangci.yml`, or equivalent configuration file that defines the exact tools and their settings. Or, add a note: "The specific toolchain is defined in the project's CI configuration."

- **Medium: Light Workflow "Required Inputs" include only "approved spec + a one-sentence approach," but spec can be in spec-light.md format.** This creates an inconsistency: the Light Workflow variant for Specify produces a one-paragraph spec, but the Execute Light Workflow expects a "one-sentence approach" as input, which is a different format. **Severity: Medium** — mismatch between phases. **Fix:** Either align the Light Workflow variants so the output of Specify Light is the same as the input to Execute Light (i.e., "one-sentence approach" is already included in the spec-light.md output), or explicitly note that Execute Light may use either the spec-light.md or the one-sentence approach.

- **Low: "No flaky tests (run test suite 3 times; results must be identical)" — ties verification to a specific run count without addressing flaky test handling.** Three runs is a heuristic, not a guarantee of non-flakiness. Also, the definition of "identical" is ambiguous (same order? same pass/fail per test?). **Severity: Low** — imprecise but unlikely to cause real problems. **Fix:** Replace with: "No known flaky tests. If a test fails intermittently, it must be quarantined, documented, and fixed before the next cycle. Run the full test suite at least once; if any failure is non-deterministic, investigate and either fix or quarantine before proceeding."

- **Low: "Maintenance" section says "It is a continuous activity that uses Light Workflow or Hotfix Workflow" but does not state that the main workflow's Discover/Specify/Design phases may still apply.** A user might assume that all maintenance follows the Light Workflow and skip problem framing or specification entirely, even for non-trivial changes. **Severity: Low** — could lead to incomplete specification for larger maintenance tasks. **Fix:** Add a note: "For maintenance tasks that add new behavior or change existing behavior beyond a simple fix, follow the Full Workflow from Discover through Decide."
