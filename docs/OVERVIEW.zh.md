# Nexus 流程概览（中文）

> 本文是流程速览。规格、设计、协议的完整定义见 `docs/` 各文件。

## Nexus 是什么

**人类与 AI 协作开发软件时的工程协调层**。不是 AI 框架、不是自动化平台、不是提示词库。

核心假设：**AI 是不稳定的概率系统**。Nexus 不试图让 AI 变稳定，而是用流程把不稳定性约束在可管理的范围内。

---

## 三个公理

| 公理 | 实践要求 |
|------|----------|
| **AI 是概率系统** | 关键结果必须验证，绝不假设一次输出就正确 |
| **上下文是有限资源** | 每个会话显式加载上下文；大任务拆分 |
| **人类判断是最终的** | AI 提议，人类决定范围、架构、验收 |

## 五条绝对规则

1. **规格 > 提示词** — 没有规格引用，不生成代码
2. **流程 > 聊天** — 一个会话服务一个阶段、一个任务
3. **审查 > 生成** — 机器查类型/测试/安全，人类查架构/逻辑
4. **显式上下文 > 隐式记忆** — 会话开始时显式加载
5. **人类最终判断** — AI 提议，人类决定

---

## 生命周期

六个阶段顺序执行，可回退，**不可跳过**：

1. **Discover** — 发现问题
2. **Specify** — 编写规格
3. **Design** — 技术设计
4. **Execute** — 实现任务
5. **Verify** — 验证测试
6. **Decide** — 决策交付

**Maintenance**（维护）是持续活动，使用 Light / Hotfix 流程。

```mermaid
flowchart LR
    A[Discover] --> B[Specify] --> C[Design] --> D[Execute] --> E[Verify] --> F[Decide]
    F -. 新问题 .-> A
```

## 各阶段速览

每个阶段的退出条件都是运行对应的 `CHECKLIST: <阶段>`（见 `docs/CHECKLISTS.md`）。

### 1. Discover — 发现问题

| | 内容 |
|--|--|
| **目的** | 在定义解决方案之前先理解问题 |
| **人类** | 描述问题、决定什么重要、拒绝范围蔓延 |
| **AI** | 提问澄清、暴露隐藏假设、精确总结 |
| **退出** | 人类确认问题值得解决且范围足够窄 |

### 2. Specify — 编写规格

| | 内容 |
|--|--|
| **目的** | 把问题写成可签字的合同 |
| **人类** | 决定范围、验证验收标准、明确批准 |
| **AI** | 结构化规格、显式化假设、建议非目标 |
| **强制章节** | Problem / Scope / Non-Goals / Acceptance Criteria / Assumptions / Dependencies / Open Questions（详见 `docs/templates/spec.md`） |
| **退出** | `CHECKLIST: Specify` 通过 |

> 多特性场景：先用 `FEATURE_ROADMAP.md` 分解 → 排优先级（P0–P3）→ 划 MVP 边界。

### 3. Design — 技术设计

| | 内容 |
|--|--|
| **目的** | 把规格变成可执行的计划 |
| **人类** | 选架构、验证接口、确认任务分解、批准 |
| **AI** | 至少两个架构选项+权衡、设计契约、拆分任务 |
| **核心产出** | `DESIGN_DOC.md`、接口契约（无 `any`/占位符）、任务（≤2h）、能力评估（High/Medium/Low） |
| **退出** | `CHECKLIST: Design` 通过 |

### 4. Execute — 实现任务

| | 内容 |
|--|--|
| **目的** | 一次实现一个边界明确的任务 |
| **核心原则** | **一个会话 = 一个任务**。每次重新加载上下文 |
| **上下文限制** | ≤800 行理想，1200 行硬上限。超出就拆任务 |
| **人类** | 分配任务、审查代码、决定提交 |
| **AI** | 一次一任务、测试与代码同写、解释偏差 |
| **审查循环** | AI 输出 → 人类按清单审查 → 通过则提交，不通过则给具体反馈，AI 修改重审 |
| **Fallback Rule** | AI 对同一任务连错两次，**第三次也不会成功** → 转人工 |
| **退出** | 每任务运行 `CHECKLIST: Execute` |

会话上下文包格式见 `docs/protocols/context-packaging.md`。

### 5. Verify — 验证测试

| | 内容 |
|--|--|
| **目的** | 确认真的对，而不是"看起来对" |
| **人类** | 运行测试、手动验证 UI、判断 bug 是否阻塞 |
| **AI** | 自动化测试、补边界用例、调试 |
| **测试要求** | 单元测试全过、关键路径集成测试、UI 人工看过、边界情况覆盖、无 flaky 测试 |
| **常见陷阱** | 只测 happy path；忽略 flaky 测试 |
| **退出** | `CHECKLIST: Verify` 通过 |

### 6. Decide — 决策交付

| | 内容 |
|--|--|
| **目的** | 关闭循环，记录决策 |
| **人类** | 读验证结果、手动验关键路径、写下 go/no-go |
| **AI** | 对照规格审查实现、识别偏差和技术债务 |
| **验收** | 与规格一致；每个验收标准有证据；无关键技术债务（定义见 `docs/protocols/definitions.md`） |
| **退出** | `CHECKLIST: Decide` 通过；"部署" = 合并到 main |

## 三种工作流变体

| 变体 | 使用场景 | 文档 | 审查 |
|------|----------|------|------|
| **Full** | 新特性、架构变更、高风险、范围不清 | 完整规格 + 设计 + 任务分解 | 各阶段跑对应检查清单 |
| **Light** | 小改动、边界明确、低风险、依赖更新 | `spec-light.md` + `design-light.md` | 自动化检查 + 人类扫描 diff |
| **Hotfix** | 紧急最小修复、已知问题已知解法 | bug 报告即需求 | 已有测试通过 + 人类验证修复 |

工作流变体不是偏好而是分流决策，规则见 `docs/protocols/triage.md`。

## 可追溯性

AI 输出不可复现，但**过程必须可追溯**。每个会话结束后保存快照：

- `context.md`（会话开始时的上下文包副本）
- 规格/设计当时的副本
- AI 输出 + 代码 diff
- 人类审查结论：approved / rejected / approved with notes
- 模型 ID、Claude Code 版本、日期

规则：**永远复制不链接、一会话一快照、只追加不修改**。

## 人类与 AI 分工总览

| 阶段 | 人类 | AI |
|------|------|-----|
| Discover | 描述问题、决定重要性 | 提问澄清、精确总结 |
| Specify | 决定范围、验证标准、批准 | 结构化、暴露假设 |
| Design | 选架构、验证接口、批准 | 提选项、设计契约、拆任务 |
| Execute | 分配任务、审代码、决定提交 | 写代码和测试、解释偏差 |
| Verify | 手动验 UI、判断 bug 严重性 | 跑自动化、补边界用例 |
| Decide | 验关键路径、go/no-go、合并 | 审查实现 vs 规格 |
| Maintenance | 分流、定优先级、验修复 | 分析根因、最小修复 |

## 关键交互原则

1. **AI 从不代替人类做决定** — 可以"建议"，不能"批准"
2. **每个会话开始时重新加载上下文** — 不要假设 AI 记得昨天
3. **给具体反馈，不是模糊指令** — 不说"修一下"，说"第 42 行边界检查遗漏负数"
4. **任务太大就拆分** — 上下文超 800 行 = 任务太大
5. **文档是记忆，不是辅助** — 快照才是记忆，不是"AI 应该记得"
