# Nexus 流程概览（中文）

## Nexus 是什么

Nexus 不是 AI 框架，不是自动化平台，也不是提示词库。它是**人类与 AI 协作开发软件时的工程协调层**。

它的核心假设只有一个：**AI 是不稳定的概率系统**。相同的输入可能产生不同的输出，AI 会遗忘、会出错、会"幻觉"。Nexus 不试图让 AI 变稳定，而是通过流程、文档和检查点，把不稳定性约束在可管理的范围内。

---

## 三个基本公理

| 公理 | 含义 | 实践要求 |
|------|------|----------|
| **AI 是概率系统** | 相同输入，不同输出。没有真正的推理，只有模式匹配。 | 绝不假设一次输出就正确。关键结果必须验证。 |
| **上下文是有限资源** | AI 会遗忘、会截断、没有跨会话记忆。 | 每个会话开始时显式加载上下文。大任务拆分。 |
| **人类判断是最终的** | AI 可以提议、起草、分析，但不能做最终决定。 | 人类决定范围、选择架构、批准实现、签署验收。 |

---

## 五条绝对规则

1. **规格 > 提示词** — 没有规格引用，不生成代码。
2. **流程 > 聊天** — 每个会话服务一个阶段、一个任务。
3. **审查 > 生成** — 机器检查类型/测试/安全，人类审查架构/逻辑/可维护性。
4. **显式上下文 > 隐式记忆** — 会话开始时显式加载上下文包。
5. **人类最终判断** — AI 提议，人类决定。

---

## 生命周期总览

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ Discover │───→│ Specify  │───→│  Design  │───→│ Execute  │───→│  Verify  │───→│  Decide  │
│  发现问题  │    │  编写规格  │    │  技术设计  │    │  实现任务  │    │  验证测试  │    │  决策交付  │
└──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘
      ↑                                                                            │
      └────────────────────── 维护（Maintenance）────────────────────────────────────┘
```

阶段是顺序的。可以回退，**不可跳过**。

## 第一阶段：Discover（发现问题）

**目的**：在定义解决方案之前，先理解问题。

### 人类做什么
- 描述遇到的问题或需求
- 确认"真正的问题"是什么
- 决定哪些事情重要、哪些不重要
- **拒绝范围蔓延**："做得更好"不是问题陈述

### AI 做什么
- 提出澄清问题，挖掘隐藏假设
- 暴露边界情况和极端场景
- 用精确的语言总结问题

### 输入与输出
| 输入 | 输出 |
|------|------|
| 用户请求或问题描述 | 清晰的问题陈述 |
| 现有产品背景（如有） | 约束和假设 |
| | 未知事项清单 |
| | 初步范围边界 |

### 退出条件
人类确认：**这个问题值得解决，且范围足够窄，可以进入规格编写。**

---

## 第二阶段：Specify（编写规格）

**目的**：把问题转化为书面规格（Spec），成为后续所有工作的合同。

### 功能分解（多特性时）

如果问题包含多个用户可见的能力，先分解：

1. **列出特性** — 每个特性是一个连贯的用户可见能力（如"用户可以用 OAuth 登录"）
2. **映射依赖** — 哪些特性需要其他特性先存在？
3. **分配优先级**
   - **P0**：基础/架构。被其他一切阻塞。
   - **P1**：核心 MVP 价值。没有它产品不完整。
   - **P2**：增强。MVP 没有它也能工作。
   - **P3**：锦上添花。推迟。
4. **解决平局**：依赖优先 → 风险优先（未知技术先做，快速失败）→ 价值密度优先
5. **划定 MVP 边界** — 最小的能交付完整用户故事的特性集合

**输出**：`FEATURE_ROADMAP.md`（特性路线图）。每个特性后续有自己的 `PRODUCT_SPEC.md`。

### 人类做什么
- **决定范围**：什么做、什么不做
- **验证验收标准**：每个标准都能回答"怎么知道这件事做完了？"
- **批准规格**：明确说出"这个规格我批准了"

### AI 做什么
- 结构化规格文档，填充缺失的章节
- 把隐式假设变为显式
- 建议范围边界和非目标（Non-Goals）

### 规格文档的强制章节
1. **Problem**（问题）— 一句话说清楚要解决什么
2. **Scope**（范围）— 具体包含什么
3. **Non-Goals**（非目标）— 明确声明不做什么
4. **Acceptance Criteria**（验收标准）— 可验证的完成条件，禁止模糊词（"好"、"快"、"用户友好"）
5. **Assumptions**（假设）— 如果假设不成立，规格可能需要修改
6. **Dependencies**（依赖）— 需要哪些外部条件
7. **Open Questions**（待解决问题）— 每个问题必须有负责人

### 退出条件
运行 **CHECKLIST: Specify**。未通过前不得进入 Design。

---

## 第三阶段：Design（技术设计）

**目的**：把规格变成可执行的计划。

### 人类做什么
- **选择架构**：AI 提出选项和权衡，人类做决定
- **验证接口契约**：类型、API 模式、数据模型必须精确
- **确认任务分解**：确保没有遗漏
- **批准设计**

### AI 做什么
- 提出至少两个架构选项，附带权衡分析
- 设计接口契约（类型、函数签名、数据模型）
- 识别技术风险
- 把实现拆分为任务

### 核心产出
| 产出 | 说明 |
|------|------|
| `DESIGN_DOC.md` | 设计文档 |
| 接口契约 | 类型、API 模式、数据模型 — 禁止使用 `any` 或占位符 |
| 任务分解 | 每个任务估计不超过 2 小时 |
| 能力评估 | 对每个任务评估 AI 能力（高/中/低），定义人机分工 |

### 能力评估（关键概念）

每个任务必须评估：AI 能否独立完成？

| 能力等级 | 含义 | 分工 |
|----------|------|------|
| **High** | AI 可以独立完成，人类只需审查 | AI 实现，人类审查 |
| **Medium** | AI 可以起草，但需要人类验证关键部分 | AI 起草，人类补全关键逻辑 |
| **Low** | AI 可能会严重出错，或涉及安全/架构决策 | 人类实现，AI 只辅助子任务 |

### 退出条件
运行 **CHECKLIST: Design**。未通过前不得进入 Execute。

---

## 第四阶段：Execute（实现任务）

**目的**：一次实现一个边界明确的任务。

### 核心原则

**一个会话 = 一个任务**。不要在同一个会话中开始新任务而不重新加载上下文。如果一个任务需要多个会话，显式拆分为子任务。

### 会话启动：上下文打包（Context Package）

每个 AI 会话开始前，人类必须显式提供：

```
Phase: Execute
Task: T1 — 实现 DomainValidator.validate()

Spec (相关部分):
- AC-3: 域名必须符合 RFC 1035
- AC-4: 无效域名返回 400 错误码

Design (相关部分):
- DomainValidator.validate(name: string): Result<Domain, ValidationError>
- 使用 src/types/index.ts 中的 Result 类型

Existing code:
[粘贴 Result 类型定义]
[粘贴现有验证器模式]

Constraints:
- 不使用外部库，只用正则
- 必须处理国际化域名（punycode）

What to do:
实现 DomainValidator.validate() 并附带完整测试。
```

**关键规则**：
- 粘贴代码，不要只引用文件名（"像 UserService 那样做"是不够的）
- 上下文控制在 **800 行以内理想，1200 行绝对上限**
- 最后一条必须是本次会话的明确指令

### 会话中的审查循环

```
┌─────────────────┐
│  AI 生成一个产出物  │
└────────┬────────┘
         ▼
┌─────────────────┐
│ 人类对照检查清单审查 │
└────────┬────────┘
         ▼
   ┌────────────┐
   │  通过？     │
   └─────┬──────┘
    否 /    \ 是
    ▼        ▼
┌────────┐  ┌─────────────┐
│人类给出 │  │ 提交代码     │
│具体反馈 │  │ 标记任务完成  │
│（不是"修一下"）│  │ 进入下一个任务 │
└───┬────┘  └─────────────┘
    │
    └────→ AI 修改 → 重新审查
```

### 会话的退出条件

| 退出类型 | 条件 | 下一步 |
|----------|------|--------|
| **正常** | 产出物通过审查 | 提交，标记任务完成 |
| **规格歧义** | 两轮迭代后 AI 仍误解需求 | 停止会话，修订规格，新开会话 |
| **Fallback Rule** | AI 对同一任务连续两次产生结构性错误代码 | 停止会话，转人工实现 |
| **人类疲劳** | 单任务超过 3 轮审查仍未收敛 | 停止，记录阻塞点，升级至规格或设计修订 |

### Fallback Rule（回退规则）

如果 AI 对同一任务连续两次产生结构性错误（如类型错误、逻辑错误、安全漏洞），**第三次也不会成功**。此时：
1. 停止 AI 实现会话
2. 转人工实现
3. 重新评估剩余任务
4. AI 只用于子任务

### 人类做什么
- 把任务分配给 AI 会话（理想情况：一个任务一个会话）
- 审查生成的代码是否符合设计
- 运行测试、手动验证
- **决定何时提交**
- **Fallback Rule 触发时，果断转人工**

### AI 做什么
- 一次实现一个任务
- 测试与代码同时写（不是写完代码再补测试）
- 解释与设计的偏差
- 标记规格或设计是否需要修订

### 退出条件
每个任务运行 **CHECKLIST: Execute**。所有任务完成后才能进入 Verify。

---

## 第五阶段：Verify（验证测试）

**目的**：确认工作确实是正确的，而不是"看起来对了"。

### 人类做什么
- 运行完整测试套件
- 对 UI/UX 进行手动验证（AI 测不了界面看起来对不对）
- 验证每个验收标准是否被满足
- 决定发现的 bug 是否阻塞交付

### AI 做什么
- 运行自动化测试
- 为边界情况提出额外的测试用例
- 帮助调试失败的测试
- 验证验收标准的覆盖率

### 测试要求

| 测试类型 | 要求 |
|----------|------|
| 单元测试 | 全部通过 |
| 集成测试 | 关键路径必须通过 |
| 手动验证 | 所有 UI/UX 验收标准必须人工看过 |
| 边界情况 | 空输入、最大值、无效格式、网络故障 |
|  flaky 测试 | 运行 3 次，结果必须一致 |

### 常见陷阱
- **只测 happy path**：AI 倾向于测试"能工作"的情况。人类必须推动测边界。
- **忽略 flaky 测试**：一个时通时不通的测试比没有测试还糟。

### 退出条件
运行 **CHECKLIST: Verify**。未通过前不得进入 Decide。

---

## 第六阶段：Decide（决策交付）

**目的**：关闭循环，记录决策。

### 人类做什么
- 阅读验证结果
- 手动验证关键路径
- **做出明确的 go / no-go 决策**（必须写下来）
- 如果 go：合并到 main 分支。如果 no-go：返回 Execute。

### AI 做什么
- 对照规格和设计审查实现
- 识别偏差、遗漏或技术债务
- 总结"实际构建了什么" vs "最初请求了什么"

### 验收标准
- 实现与批准的规格一致（无静默范围变更）
- 每个验收标准都有证据（测试输出、截图、日志）
- 设计偏差有文档记录和正当理由
- 无关键技术债务（定义见 `docs/protocols/definitions.md` §Critical Technical Debt）
- 人类手动验证了关键路径

### 退出条件
运行 **CHECKLIST: Decide**。未通过前不得部署。

**"部署"的定义**：合并到 main 分支。生产发布是独立的运营问题。

## 维护（Maintenance）

维护不是主流程的一个阶段，而是**持续活动**，使用 Light Workflow 或 Hotfix Workflow。

### 人类做什么
- 分流 bug 和功能请求
- 决定现在修还是以后修
- 验证 AI 提出的修复方案

### AI 做什么
- 分析 bug 报告，建议根因
- 提出最小化修复方案
- 更新文档

### 规则
- 即使是一行修复，也必须引用 issue
- 代码变更时，文档必须同步变更
- 定期处理技术债务

## 三种工作流变体

| 变体 | 使用场景 | 文档要求 | 审查要求 |
|------|----------|----------|----------|
| **Full Workflow**（完整流程） | 新特性、架构变更、高风险任务、范围不清晰 | 完整规格 + 完整设计 + 任务分解 | 每阶段运行对应检查清单 |
| **Light Workflow**（轻量流程） | 小改动、边界明确的任务、低风险工作、依赖更新 | `spec-light.md`（一段问题描述 + 2-3 个验收标准）+ `design-light.md`（一段方法 + 接口草图） | 自动化检查通过 + 人类快速扫描 diff |
| **Hotfix Workflow**（热修流程） | 紧急最小修复、范围狭窄、已知问题已知解法 | bug 报告即需求：复现步骤、预期 vs 实际、影响范围、回归风险 | 现有测试通过 + 人类验证修复 |

**工作流变体不是偏好，而是分流决策**。根据 `docs/protocols/triage.md` 的分流规则选择。

## 可追溯性机制

AI 的输出不可复现（相同的提示词下次可能产生不同的 tokens），但**过程必须可追溯**。

### 状态快照（State Snapshot）

每个会话结束后保存：

```
sessions/
  2026-05-12-feature-name/
    02-specify/
      context.md      # 会话开始时加载的上下文包
      spec-v1.md      # 会话开始时的规格副本
      output.md       # AI 输出
      diff.patch      # 代码变更
      verdict.md      # 人类审查结论
```

**规则**：
- 永远复制，不链接。规格变更不影响历史快照。
- 一个会话一个快照。
- 快照只追加，不修改。
- 每个快照必须有书面结论：approved / rejected / approved with notes。

### 环境锁定

记录每次会话使用的模型和参数：
- 模型 ID（如 `claude-sonnet-4-6`）
- Claude Code 版本
- 日期
- 上下文窗口限制

## 人类与 AI 的协作模式总结

| 活动 | 人类 | AI |
|------|------|-----|
| **发现问题** | 描述问题、决定重要性 | 提问澄清、总结精确陈述 |
| **编写规格** | 决定范围、验证标准、批准 | 结构化文档、暴露假设 |
| **技术设计** | 选择架构、验证接口、批准 | 提出选项、设计契约、拆分任务 |
| **实现代码** | 分配任务、审查代码、运行测试、决定提交 | 写代码和测试、解释偏差 |
| **验证测试** | 手动验证 UI、决定 bug 是否阻塞 | 运行自动化测试、提出边界用例 |
| **决策交付** | 验证关键路径、做 go/no-go 决策、合并 | 审查实现 vs 规格、总结偏差 |
| **维护修复** | 分流、决定优先级、验证修复 | 分析根因、提出最小修复 |

### 关键交互原则

1. **AI 从不代替人类做决定**。AI 可以"建议"，但不能"批准"。
2. **每个会话开始时重新加载上下文**。不要假设 AI 记得昨天聊过什么。
3. **给具体反馈，不是模糊指令**。不说"修一下"，说"第 42 行的边界检查遗漏了负数情况，应该返回 ValidationError"。
4. **任务太大就拆分**。上下文超过 800 行 = 任务太大。
5. **频繁提交**。最大未提交风险是一个任务。
6. **文档是记忆，不是辅助**。"Claude 应该记得昨天的事" — 不，快照才是记忆。
